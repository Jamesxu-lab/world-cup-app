# 晨报球搭子 — MVP 技术方案与开发计划

> 版本：v1.0 | 日期：2026-05-26 | 状态：已确认

---

## 一、产品概述

### 1.1 一句话定位

给睡过比赛的中国泛球迷，用3分钟讲好昨晚发生了什么。

### 1.2 核心用户场景

2026美加墨世界杯比赛时间在北京时间凌晨3点至上午9点，绝大多数中国泛球迷赶不上直播。早上醒来打开手机，需要快速知道"昨晚那场球发生了什么"，而且要读得下去、看得有味。

### 1.3 竞品差异化

| 维度 | OddsFlow | 晨报球搭子 |
|------|----------|-----------|
| 核心价值 | 辅助投注决策 | 赛事内容体验 |
| 产出形态 | 概率/赔率表格 | 叙事卡片 + 数据可视化 |
| 用户心智 | "这场怎么买" | "这场发生了什么" |
| 语言/文化 | 英文、通用 | 中文、带梗、有情绪 |
| 社区 | 全球聊天 | 暂不做（V1） |

### 1.4 MVP范围

**做：**
- 按日期展示"今早结束的比赛"，每场带比分 + 一句话钩子
- 每场比赛5-8张叙事卡片，三种风格可选：正经复盘 / 段子手 / 战术党
- 用户可在阅读中随时切换风格
- Web H5，移动优先

**不做（V1）：**
- 实时比分推送
- 赔率/预测
- 社区/UGC/评论
- 追问对话（V1.5小组赛阶段补）
- 音频TTS（V2）
- 独立App

## 三、技术选型

### 3.1 总览

| 层级 | 技术 | 理由 |
|------|------|------|
| 前端框架 | React 18 + TypeScript | 生态成熟、开发效率高 |
| 样式 | Tailwind CSS | 移动优先、原子化CSS |
| 状态管理 | Zustand | 轻量、够用 |
| 图表 | Recharts | 轻量React图表库 |
| 构建工具 | Vite | 开发体验好、构建快 |
| 后端框架 | FastAPI (Python 3.12) | 异步、高性能、自动API文档 |
| 数据库 | PostgreSQL | 结构化赛事和内容数据 |
| 向量数据库 | ChromaDB | 轻量、Python原生 |
| 缓存 | Redis | 热点内容缓存、Celery broker |
| 任务队列 | Celery | AI生成异步任务 |
| LLM | OpenAI 兼容 API | 用户已有配置 |
| 赛事数据 | API-Football v3 | 全量事件流 + 技术统计 |
| ORM | SQLAlchemy 2.0 + Alembic | 数据库迁移管理 |
| 部署 | Docker Compose + Nginx | 一键部署、标准化 |

### 3.2 为什么选 API-Football v3

| 维度 | API-Football v3 |
|------|----------------|
| 比赛事件流 | 全量（进球/红黄牌/换人/点球/VAR/助攻） |
| 技术统计 | 射门/xG/控球/传球成功率/角球/跑动距离 |
| 球员数据 | 评分/进球/助攻/传球/抢断 |
| 阵容 | 首发/替补/阵型 |
| 世界杯覆盖 | 全覆盖 |
| 免费额度 | 100次/天（小组赛测试够用） |
| 付费计划 | $75-150/月（淘汰赛升级） |

### 3.3 核心数据模型

```sql
-- 比赛表
matches (
    id UUID PRIMARY KEY,
    fixture_id INTEGER UNIQUE,         -- API-Football 比赛ID
    home_team VARCHAR(100),
    away_team VARCHAR(100),
    home_score INTEGER,
    away_score INTEGER,
    match_date TIMESTAMP,
    group_name VARCHAR(50),
    round VARCHAR(50),                 -- group_stage / round_of_32 / ...
    stadium VARCHAR(200),
    city VARCHAR(100),
    status VARCHAR(20),                -- scheduled / live / finished
    created_at TIMESTAMP DEFAULT NOW()
);

-- 比赛事件表（原始数据）
match_events (
    id UUID PRIMARY KEY,
    match_id UUID REFERENCES matches(id),
    minute INTEGER,
    extra_minute INTEGER,
    event_type VARCHAR(30),            -- goal / card / substitution / var
    detail VARCHAR(200),               -- 具体描述
    player_name VARCHAR(100),
    team VARCHAR(100),
    assist_player VARCHAR(100),
    created_at TIMESTAMP DEFAULT NOW()
);

-- 比赛统计表
match_stats (
    id UUID PRIMARY KEY,
    match_id UUID REFERENCES matches(id),
    team VARCHAR(100),
    stat_type VARCHAR(50),             -- shots_on_target / possession / xg / ...
    stat_value NUMERIC
);

-- 球员表现表
player_performances (
    id UUID PRIMARY KEY,
    match_id UUID REFERENCES matches(id),
    player_name VARCHAR(100),
    team VARCHAR(100),
    position VARCHAR(50),
    rating NUMERIC(4,2),
    goals INTEGER DEFAULT 0,
    assists INTEGER DEFAULT 0,
    stats_json JSONB                   -- 详细统计数据
);

-- 叙事内容表（核心产出）
narratives (
    id UUID PRIMARY KEY,
    match_id UUID REFERENCES matches(id),
    style VARCHAR(20),                 -- formal / funny / tactical
    card_index INTEGER,                -- 卡片序号 1-8
    card_type VARCHAR(30),             -- overview / key_moment / player / stats / ...
    title VARCHAR(200),
    content TEXT,
    model_version VARCHAR(20),
    generated_at TIMESTAMP DEFAULT NOW(),
    UNIQUE(match_id, style, card_index)
);
```

## 四、AI叙事引擎设计（核心模块）

### 4.1 处理流水线

```
API-Football 原始数据（事件流 JSON + 统计数据）
        │
        ▼
┌─────────────────────────┐
│ Step 1: 数据预处理       │  Python 脚本
│  - 提取关键事件           │  - 筛选进球/红牌/争议判罚等关键事件
│  - 排序 + 时间线构建      │  - 构建 "比赛时间线摘要"
│  - 统计聚合              │  - 控球率/xG/射门分布等关键数据
│  - 球员表现TOP5          │  - 评分最高/最低球员
└────────┬────────────────┘
         │ 结构化中文摘要 (dict)
         ▼
┌─────────────────────────┐
│ Step 2: 上下文增强 (RAG)  │  ChromaDB 检索
│  - 球队历史故事           │  - 检索该队历史典故/经典名场面
│  - 球员背景              │  - 明星球员的故事和风格
│  - 交锋历史              │  - 两队往届对阵记录
│  - 世界杯历史            │  - 相关世界杯历史典故
└────────┬────────────────┘
         │ 增强后的上下文
         ▼
┌──────────────────────────────────────────────┐
│ Step 3: 三路并行生成                           │
│                                               │
│  ┌───────────────┐ ┌───────────────┐ ┌──────────────┐ │
│  │ 正经复盘Prompt │ │ 段子手Prompt   │ │ 战术党Prompt  │ │
│  │ (《天下足球》) │ │ (微博热搜风)   │ │ (教练复盘)    │ │
│  └───────┬───────┘ └───────┬───────┘ └──────┬───────┘ │
│          ▼                 ▼                 ▼         │
│      LLM 调用          LLM 调用          LLM 调用      │
│                                               │
│  输出: 5-8张叙事卡片 × 3种风格                  │
└──────────────────────────────────────────────┘
         │
         ▼
┌─────────────────────────┐
│ Step 4: 质量校验          │  Python 后处理
│  - 事实核查              │  - 比分/球员名/事件一致性检查
│  - 长度校验              │  - 每张卡片 100-200 字
│  - 敏感词过滤            │  - 政治/歧视/暴力内容拦截
│  - 去重检查              │  - 三张卡片之间内容不重复
└────────┬────────────────┘
         │ 通过校验
         ▼
    入库 (narratives 表)
```

### 4.2 三种风格 Prompt 设计

#### 正经复盘（近似《天下足球》画外音）

```
你是一位资深足球评论员，文风接近《天下足球》画外音。

请基于以下比赛数据，生成 {card_count} 张叙事卡片。

要求：
- 开头卡片：用一句感性总结或经典语录开篇
- 关键事件卡片：按时间线描述每个转折点，带人物弧光
- 球员卡片：突出1-2位关键球员的表现，有细节
- 数据卡片：引用关键数据，但用叙事语言包装
- 结尾卡片：呼应开头，留下余味
- 每张卡片 150-200 字
- 语言风格：正式但不枯燥，有情感但不煽情

比赛数据：
{structured_match_summary}
{rag_context}
```

#### 段子手（微博/虎扑步行街风）

```
你是一个懂球的段子手，在虎扑步行街混了十年。

请基于以下比赛数据，生成 {card_count} 张叙事卡片。

要求：
- 用网络化语言，敢调侃，敢吐槽
- 抓住比赛中戏剧性的瞬间放大
- 给表现差的球员/教练起外号（友好范围）
- 带中国球迷视角——"天台见"级别的情感共鸣
- 可以拉踩，但要有道理
- 每张卡片 100-150 字
- 语言风格：幽默、犀利、有网感

比赛数据：
{structured_match_summary}
{rag_context}
```

#### 战术党（教练席视角）

```
你是一位战术分析师，曾在职业俱乐部工作。

请基于以下比赛数据和技术统计，生成 {card_count} 张叙事卡片。

要求：
- 关注阵型变化和关键战术博弈
- 引用 xG、传球网络、跑动热区、高位逼抢等专业概念
- 解释"为什么赢/输"，而非仅仅描述
- 指出战术胜负手——教练的关键调整
- 用专业术语但保持可读性
- 每张卡片 150-200 字
- 语言风格：专业、冷静、有洞察

比赛数据：
{structured_match_summary}
{rag_context}
技术统计：
{match_stats_detail}
```

### 4.3 RAG 知识库构建

| 知识维度 | 数据来源 | 处理方式 | 预计条目 |
|---------|---------|---------|---------|
| 48支球队简介 | Wikipedia API | 抓取 → 分块 → ChromaDB | ~48条 |
| 明星球员档案（Top100） | Wikipedia API + 新闻 | 抓取 → 分块 → ChromaDB | ~100条 |
| 世界杯历史典故 | 手工整理中文资料 | 写入 → ChromaDB | ~30-50条 |
| 球队交锋史 | API-Football 历史数据 | 脚本批量导入 | ~100条 |
| 经典名场面 | 手工整理（1:7/上帝之手等） | 写入 → ChromaDB | ~20条 |

**MVP策略**：先覆盖淘汰赛热门8队（巴西/阿根廷/法国/英格兰/德国/西班牙/葡萄牙/荷兰），小组赛阶段逐步补全其他球队。

### 4.4 追问对话（V1.5，小组赛中后期补）

```
用户提问："内马尔这场表现怎么样？"
        │
        ▼
┌───────────────────┐
│ 意图识别           │  → class Intent(str, Enum): PLAYER / TACTICS / EVENT / CHITCHAT
└───────┬───────────┘
        │
        ▼
┌───────────────────┐
│ 上下文组装         │  → 本场比赛数据 + 球员历史表现 + RAG球员档案
└───────┬───────────┘
        │
        ▼
    LLM Streaming → SSE → 前端打字动画
```

```
1. 首页 — 比赛列表

   │   ├── 卡片类型图标 + 标题
```

### 5.2 移动优先设计规范

- 卡片宽度 100%，最大 480px 居中
- 正文字号 16px，标题 18-20px
- 触控热区 ≥ 44px × 44px
- 支持系统深色模式（prefers-color-scheme）
- 首屏加载目标 < 2秒
- 图片（队徽等）使用 WebP 格式 + 懒加载

**目标**：三种风格叙事能从原始数据自动生成，质量达到"能读"标准。

```
□ Day 3-4：数据预处理器
  ├── 事件提取 + 排序 + 时间线构建
  ├── 统计聚合（控球/xG/射门分布等）
  ├── 球员表现TOP5提取
  └── 输出 "结构化中文摘要" 中间格式

□ Day 4-5：Prompt 工程
  ├── 正经复盘 Prompt 模板 + 3场测试迭代
  ├── 段子手 Prompt 模板 + 3场测试迭代
  ├── 战术党 Prompt 模板 + 3场测试迭代
  └── 卡片切分逻辑（保证5-8张，不重复）

□ Day 5-6：RAG 知识库
  ├── 8支热门球队简介写入 ChromaDB
  ├── Top 50 球员档案写入
  ├── 20条世界杯历史典故写入
  └── 检索 + 注入流水线跑通

□ Day 6-7：质量校验层
  ├── 事实一致性检查（比分/球员/事件交叉比对）

  ├── 卡片长度校验

```

**交付物**：对任意一场比赛，能自动生成三种风格、5-8张叙事卡片，内容质量通过人工验收。

**这是整个项目成败的关键阶段**。如果AI生成的内容读起来像"填空模板"，产品就没有灵魂。

### Phase 2：后端 API + 自动生成（Day 8-10，6/3-6/5）

**目标**：前端能调API拿到比赛列表和叙事内容。

```
□ FastAPI 路由开发
  ├── GET /api/v1/matches?date=YYYY-MM-DD     → 比赛列表 + 钩子
  ├── GET /api/v1/matches/{id}                 → 比赛详情
  ├── GET /api/v1/matches/{id}/narrative?style=funny → 叙事卡片
  ├── GET /api/v1/teams/{id}                   → 球队信息
  └── CORS 配置 + 错误处理

□ Celery 异步任务
  ├── 赛后触发脚本（监控比赛状态变化）
  ├── 数据拉取 + 预处理 + AI生成 串行任务
  └── 生成失败自动重试 + 手动触发接口

□ Redis 缓存策略
  ├── 比赛列表缓存（TTL: 10分钟）
  ├── 叙事内容缓存（永久，生成后不改）
  └── 缓存失效机制

□ 管理接口
  ├── POST /api/v1/admin/regenerate/{match_id} → 手动重新生成
  └── GET  /api/v1/admin/generation-status     → 查看生成队列状态
```

**交付物**：API可用，一场比赛从数据拉取到叙事生成全自动。

## 八、关键风险与缓解

| 风险 | 严重度 | 缓解策略 |
|------|--------|---------|
| AI叙事质量不稳定 | 🔴 致命 | Phase 1 测试10+场历史比赛；建立质量评分卡；Prompt模板化降低波动 |

| 16天开发时间不够 | 🟠 高 | 每日standup；优先保证 AI 引擎 + 前端叙事页，其余可降级 |
| 数据源不稳定 | 🟠 高 | 备用 Football-Data.org 免费源；关键字段交叉校验 |
| 小组赛冷门比赛没人看 | 🟢 低 | 小组赛阶段人工选择热门比赛生成，无需全量 |
| LLM API 成本超预期 | 🟡 中 | 生成立即缓存，不改动不重新生成；监控每日 token 消耗 |
| 版权/合规 | 🟡 中 | 仅使用结构化数据生成原创叙事；RAG素材来自 Wikipedia (CC协议) |
